1. 무엇을 만들었나

검색 품질을 감이 아니라 숫자로 확인하는 어드민 도구. 헬프센터/검색팀(Google·Algolia·Zendesk)이 쓰는 표준 지표를 도입.

Layer성격지표위치
A. 오프라인 평가골든셋(정답셋) 회귀 테스트 — 트래픽 불필요, 즉시 실행Hit@1 · Hit@3 · MRR · nDCG@5/admin/search-quality
B. 온라인 지표실사용 로그 — 데이터 누적0건율 · 0건 top질의 · CTR · 평균 클릭위치 · 검색→접수 전환율 · 자가해결(deflection)/admin/search-quality/usage

2. 지표 정의 (사장님 "100문항 top3" 아이디어의 정식화)

지표의미
Hit@3정답이 top3 안에 든 질의 비율 — "100문항 중 몇 %가 top3"가 이 수치 (목표 ≥90%)
Hit@1정답이 1위인 비율
MRR정답 평균 등수의 역수 (1위=1.0, 2위=0.5, 3위=0.33)
nDCG@5위치·등급 가중 랭킹 품질 (구글/아마존 검색팀 표준)
deflection검색 후 접수로 안 넘어간 비율 = 자가해결 추정 (Zendesk/Intercom 핵심 KPI)

3. 골든셋 채우는 3가지 방법

방법설명
FAQ 시드기존 FAQ가 이미 "질문→정답" 쌍 — 버튼 한 번에 골든셋화 (멱등)
AI 질의 생성gpt-4o-mini가 아티클을 읽고 현실적 사용자 질문 생성 (구어·증상형)
수기 등록자주 묻는 100문항을 직접 등록 (질의 + 정답 slug/FAQ id)
판정 모드 3종: label(정답 라벨, 빠름·무료) / llm(GPT가 적합도 0~3 채점) / hybrid(라벨 없으면 LLM). 랭킹 파라미터를 바꾸고 재실행 → 점수 증감을 전후 비교 = 검색 회귀 테스트.

4. 첫 실측 결과 (FAQ 12건, label)

Hit@1 67% · Hit@3 83% · MRR 0.78 · nDCG@5 0.81. 골든셋이 12건으로 작고, 스크립트 실행이라 동의어 캐시가 빠진 보수적 수치 — 실제 대시보드(동의어 적용)에선 같거나 더 높음. FAQ 시드 + AI 생성 + 100문항 등록으로 골든셋을 키우면 신뢰도 상승.

5. 신규 DB 테이블 (마이그레이션 0021·0022)

테이블용도
search_eval_queries골든셋 — 질의 + 정답(아티클slug/FAQid) + 출처(faq/manual/llm)
search_eval_runs평가 실행 스냅샷 — Hit/MRR/nDCG + 질의별 상세(jsonb)
search_logs실사용 검색 1행 — 결과수·0건여부·클릭·전환·제품·사용자

6. 주요 신규 파일

  • lib/services/search-eval.ts — 평가 엔진(Hit/MRR/nDCG, 통합 랭킹, 시드/생성/CRUD)
  • lib/services/search-logs.ts — 로깅 + 집계(0건율·CTR·deflection)
  • lib/services/llm.ts — gpt-4o-mini 질의 생성 + 적합도 채점 (graceful)
  • app/(admin)/admin/search-quality/ — 평가 대시보드 + 골든셋 관리 + 사용 지표
  • app/api/search/track/route.ts + app/search/_components/tracked-link.tsx — 클릭·전환 추적(sendBeacon)
  • app/actions/search-eval-actions.ts — 평가 실행·시드·생성·CRUD 서버 액션

8. ⚠️ 별도 사안 — 기존 빌드 깨짐 (제 작업과 무관)

npm run build(Turbopack)가 제 변경 전부터 이미 실패 중. clean HEAD(stash 후)에서도 동일. 원인: 기존 lib/ai/anthropic-client.ts(아티클 AI 리라이터)가 @anthropic-ai/sdk를 클라이언트 번들로 끌고 들어감(article-editor.tsx 경유). 제 신규 파일은 에러 목록에 없음. 검증은 tsc --noEmit + 평가/로그 실측 동작으로 대체. 배포 전 이 빌드 에러를 별도 수정 필요.
