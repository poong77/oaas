1. 장애 ① — ticket_no UNIQUE 충돌

증상 (Vercel 함수 로그)

[warning] [tickets.createTicket] ticket_no 충돌 재시도 1/5: AS-2026-000010
[warning] ... 재시도 2/5 ~ 5/5
[error] 재시도 한계 초과, 마지막 에러: NeonDbError 23505
  duplicate key value violates unique constraint "tickets_ticket_no_uq"
  Key (ticket_no)=(AS-2026-000014) already exists.

근본 원인

레이어문제
채번 알고리즘 MAX(ticket_no) + 1 기반. 충돌 시 bump 증분 후 재시도.
Neon 특성 read replica lag 1~2초. 동시 INSERT 요청들이 모두 stale currentMax를 읽고 같은 슬롯 시도.
당시 운영 코드 bump×1, retry=5 — 5건 동시 접수만으로 100% 충돌. 로그의 000010~000014가 정확히 5건 충돌 증거.

수정 (commit fa35d27)

MAX 기반 → atomic UPSERT 카운터 테이블. Postgres 단일 statement로 race-free 보장. replica lag 무관.

-- db/migrations/0010_red_piledriver.sql
CREATE TABLE "ticket_no_counter" (
  "year" integer PRIMARY KEY NOT NULL,
  "last_no" integer DEFAULT 0 NOT NULL
);

-- 기존 운영 ticket_no MAX로 초기화 (시드 ≥900000 제외)
INSERT INTO "ticket_no_counter" ("year", "last_no")
SELECT
  CAST(SUBSTRING("ticket_no" FROM 4 FOR 4) AS INTEGER) AS year,
  MAX(CAST(SUBSTRING("ticket_no" FROM 9) AS INTEGER)) AS last_no
FROM "tickets"
WHERE "ticket_no" ~ '^AS-[0-9]{4}-[0-9]{6}$'
  AND CAST(SUBSTRING("ticket_no" FROM 9) AS INTEGER) < 900000
GROUP BY CAST(SUBSTRING("ticket_no" FROM 4 FOR 4) AS INTEGER)
ON CONFLICT ("year") DO UPDATE
  SET "last_no" = GREATEST("ticket_no_counter"."last_no", EXCLUDED."last_no");
// lib/services/tickets.ts — generateTicketNo()
INSERT INTO ticket_no_counter (year, last_no)
VALUES (${year}, 1)
ON CONFLICT (year) DO UPDATE
  SET last_no = ticket_no_counter.last_no + 1
RETURNING last_no

운영 검증

  • 마이그레이션 직후 ticket_no_counter(2026, 20) — 실제 운영 MAX와 일치
  • 재배포 후 신규 티켓 AS-2026-000021 정상 채번
  • 연도 PK 구조라 2027-01-01 첫 티켓은 자동 AS-2027-000001

2. 장애 ② — 마스터 페이지 전수 500

증상

  • 사용자 신고: /admin/master/categories, /admin/master/quick-actions/[id] 등 전 마스터 페이지 500
  • 로컬 dev + prod build 양쪽 모두 정상 200 — 운영에서만 재현

근본 원인

Next 16 Turbopack이 'use server' 파일의 export type { CategoryType };(타입 re-export)을 런타임 value export로 잘못 emit. TS 컴파일 단계에선 strip되지만 Server Actions 번들러 경로에서 누락. 동일 파일을 import하는 모든 어드민 마스터 페이지가 module init 시점에 ReferenceError로 죽음.

왜 로컬은 멀쩡했나

로컬 next dev(Turbopack), next start(prod build) 모두 정상 작동. Vercel 함수 cold start 경로의 ESM 직렬화 순서에서만 노출되는 경계 케이스. 빌드·타입체크로는 검출 불가.

수정 (commit 1da31af)

// app/actions/master-actions.ts
- import type { CategoryType } from '@/db/schema';  // 미사용
  ...
- // 타입 export
- export type { CategoryType };                      // 원흉

외부 사용처는 모두 이미 @/db/schema에서 직접 import 중이라 영향 0.

6. 향후 권장 사항

  • Server Actions lint 룰: eslint-plugin-next 또는 자체 룰로 'use server' 파일에서 export type { ... };(re-export) 금지 자동 검출. 동일 함정 재발 방지.
  • 운영 부하 테스트: 동시 티켓 접수 50~100건 부하 테스트로 atomic counter 동작 확인 권장 (별도 작업).
  • 티켓 번호 연도 정책: 연도별 카운터 reset이 의도된 동작인지 운영팀 확인. 글로벌 누적 번호가 더 좋다면 schema PK를 (year, last_no) → 단일 last_no로 변경 가능.
  • dev-rules.md 보강: 'use server' 파일 export 제약 항목 추가 — async 함수 외에는 로컬 정의 타입만 export.

7. 학습 포인트

로컬에서 재현 안 되는 운영 장애는 stack trace 없으면 절대 못 잡는다. Vercel 에러 ID(398581919, 2128134143)는 사용자 화면용 식별자일 뿐 원인이 아니다. 함수 로그의 첫 줄 에러 메시지 + 파일:라인이 있어야 1샷에 잡힌다. 사용자에게 stack trace를 명확히 요청하는 것이 추측·소거법보다 10배 빠르다.
Race condition을 retry 횟수로 막지 마라. MAX 기반 채번은 본질적으로 race-prone. retry는 확률을 낮출 뿐 0으로 만들지 못한다. Postgres의 INSERT ... ON CONFLICT DO UPDATE RETURNING은 단일 statement atomic이라 race-free. 근본 해결이 임시 완화보다 코드량도 적다 (76 줄 → 30 줄).
