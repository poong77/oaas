🎯 4-Perspective 가치 요약

관점내용
Problem 아티클 관리 페이지 상단 "전체 (이 페이지)" 통계 카드가 현재 페이지 항목 수(20)만 표시. 헤더 설명문은 "전체 95건"인데 카드는 20 → 수치 불일치로 운영자 혼란.
Solution listArticles의 total count 쿼리에 PostgreSQL count(*) filter (...)를 추가해 발행/Draft 수를 한 번에 집계. 페이지(items) 기준 계산을 제거하고 필터 전체 기준 값으로 교체.
Function·UX (1) "전체" 카드 = 필터 조건 전체 건수(95) (2) "발행"/"Draft" 카드도 전체 기준 → 카드 합과 헤더 "전체 N건"이 항상 일치 (3) 검색·필터 적용 시에도 해당 조건 전체 기준으로 정확 집계.
Core Value 운영자가 통계 카드만 봐도 전체 콘텐츠 규모·발행 현황을 즉시 파악. 추가 쿼리 없이(기존 count 쿼리 1개 재활용) 성능 영향 없음.

💡 인사이트

"이 페이지 내 집계"는 페이지네이션 환경의 흔한 함정. 통계 카드는 사용자가 "전체 현황"으로 인식하므로 반드시 서버 측 전체 집계로 채워야 함. PostgreSQL FILTER 절을 쓰면 발행/Draft 분리 집계를 추가 쿼리 없이 total 쿼리 하나에 묶을 수 있어 성능 비용 0.
  • 리스트 서비스 반환 타입에 집계 필드를 추가할 땐 모든 early-return 경로(db null, catch)도 함께 갱신해야 tsc 통과
  • 같은 패턴을 getArticleCountsByProduct 등 다른 카운트 API에도 점검 여지 (발행/비발행 분리 필요 시)
