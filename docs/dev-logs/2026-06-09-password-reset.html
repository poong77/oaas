1. 보안 설계 결정 — 임시비번 ❌ → OTP/토큰 ✅

최초 기획은 "호텔 검색 → 임시비밀번호 발송 → 로그인 후 변경"이었으나, 다음 약점으로 더 안전한 방식을 채택했습니다.

구분임시비밀번호 (원안)OTP/토큰 (채택)
핵심 위험임시비번이 유효한 로그인 자격이 됨 (가로채기·메일함 잔존 시 장시간 악용)인증수단은 재설정 화면을 여는 열쇠일 뿐, 그 자체로 로그인 불가
이메일임시비번 평문 전송일회용 재설정 링크 (30분 만료·1회용)
문자임시비번 평문 전송6자리 인증코드 (10분 만료·5회 제한·1회용)
저장해시된 비번이 즉시 활성화토큰/코드 sha256 해시만 저장, 변경 전까지 기존 비번 유효

2. 사용자 흐름

  1. 로그인 화면 → "비밀번호를 잊으셨나요?"
  2. 호텔 검색 (국문/영문/띄어쓰기·하이픈 무시 — collapseSpacing)
  3. 해당 호텔의 본인 계정 선택 (이름·이메일·전화 모두 마스킹: 홍*동 · ma***@g***.com · 010-****-5678)
  4. 받을 방법 선택 → 이메일(링크) 또는 문자(코드)
  5. 문자: 6자리 코드 입력 검증 / 이메일: 메일의 링크 클릭
  6. 새 비밀번호 직접 설정 (8자+영문+숫자) → 완료 알림 + 감사 로그

3. enumeration 위험 완화

호텔 검색으로 마스킹 연락처를 노출하는 구조는 계정 존재가 드러나지만(호텔리어가 로그인 ID를 모르는 현실 UX 우선), 아래로 완화했습니다.

엔드포인트Rate Limit
호텔 검색 / 계정 목록IP당 30회/분
코드·링크 발송IP당 5회/분 + 계정당 3회/분
코드 검증 / 완료IP당 10회/분 + 코드 5회 오입력 시 토큰 폐기
마스킹 강화 + 감사 로그(user.password_reset_requested, user.password_self_reset) + 1회용 토큰으로 악용 시나리오를 차단합니다.

5. 배포 전 필수 작업

① 마이그레이션 적용 — 개발: npm run db:migrate / 프로덕션: drizzle-kit migrate
⚠️ db:push 금지 (검색 인덱스 DROP 위험 — 기존 정책)

② 환경변수 확인 — PUBLIC_BASE_URL(또는 NEXTAUTH_URL)이 실제 도메인이어야 재설정 링크가 올바르게 생성됩니다. SES/솔라피 키 설정 확인.

6. 검증 결과 & 수동 테스트 항목

통과 tsc --noEmit 0 에러 · next build 성공 (8개 라우트 등록)

수동 테스트(스테이징 권장):

  • 호텔명 다양한 표기(붙여쓰기/영문) 검색 정확도
  • 이메일 링크 수신 → 30분 만료/1회용 동작
  • 문자 코드 수신 → 5회 오입력 차단, 만료 동작
  • placeholder 이메일(@as.local) 계정은 이메일 채널 미노출
  • 변경 완료 후 새 비번 로그인 + must_change_password=false
