호텔리어가 셀프 픽스에서 해결되지 않은 이슈를 3단계 폼으로 접수하면, 티켓번호 자동 발급 → 본인에게 SMS/이메일 → 운영팀 Slack #as-new(P1은 #as-urgent) 알림이 자동으로 발송됩니다. 매니저는 큐에서 상태/담당/마감일 변경, 공개 답변, 내부 메모, Dev 에스컬을 처리합니다.

2신규 스키마

테이블역할주요 인덱스
tickets 이슈 티켓 본체. AS-YYYY-NNNNNN 번호, 상태/담당/마감일/연락방식/Slack 스레드ts. ticket_no UNIQUE, (status, created_at), (reporter_id, created_at), (hotel_id, created_at), (assignee_id, status), (urgency, status)
ticket_messages 스레드 메시지 — public / internal_memo / status_change / system (ticket_id, created_at), (ticket_id, kind)
ticket_attachments Vercel Blob 업로드 메타 (URL·pathname·MIME·크기). (ticket_id), (message_id)
ticket_form_fields Phase 9 어드민 편집 동적 폼 정의. 현재 Phase는 빈 테이블 (마이그레이션만). (product_code, field_key) UNIQUE
notification_logs SMS/Email/Slack 발송 이력. append-only. fire-and-forget 기록. (related_ticket_id, created_at), (status)
티켓번호 발급 — 현재 연도 기준 AS-YYYY-NNNNNN 형식. generateTicketNo()에서 MAX 조회 + retry 1회. 동시 접수 충돌 시 +N으로 자동 보정.

3알림 시스템

채널 3종

  • 솔라피 SMS — lib/notifications/solapi.ts 실 SDK 연동으로 교체 (동적 import, 키 미설정 시 stub).
  • AWS SES — 기존 모듈 유지 + notifyEmail로 wrapping하여 notification_logs 자동 기록.
  • Slack Incoming Webhook — lib/notifications/slack.ts 신규. Block Kit 빌더 3종 (new/urgent/escalate).

9메인 세션 실행 절차

이 단계는 메인 세션이 수행합니다. 본 작업자는 코드/시드 데이터만 작성. 

npm install
rm db/migrations/0003_phase5_tickets_placeholder.sql
npx drizzle-kit generate          # 0003_*.sql 자동 생성
npm run db:migrate                # 또는 dev면 npm run db:push
npm run db:seed                   # 샘플 티켓 3건 시드
npm run build                     # 타입체크 + production build
git add -A && git commit -m "feat(phase-5): 이슈 클레임 — 티켓·메시지·첨부·알림·매니저 큐"
git push

로컬 테스트 권장 시나리오

  1. 호텔리어 접수: hotelier@oa.local 로그인 → /tickets/new 3단계 폼 → 첨부 1개 업로드 → 제출 → /tickets/[id]?created=1 리다이렉트 확인.
  2. 알림 발송: 콘솔에서 SMS/Email/Slack stub 로그 확인. 실 키 설정 시 1건씩 실제 전송.
  3. 매니저 큐: manager@oa.local 로그인 → /admin/tickets → 시드 티켓 3건 + 새 티켓 1건 노출.
  4. 상태 변경: 상세에서 상태 → in_progress 변경 → status_change 메시지 자동 추가 + 호텔리어 알림 발송.
  5. 내부 메모: 매니저 화면에서 internal_memo 작성 → 호텔리어 화면 새로고침 → 메모 미노출 확인.
  6. Dev 에스컬: 사이드바에서 사유 10자 이상 입력 → 확인 → Slack #dev-escalation stub 로그 + internal_memo 자동 기록.
  7. 전화 접수: /admin/tickets/new-by-phone → 호텔 선택 → 호텔리어 선택 → 폼 제출 → channel='phone' 저장 확인.

10알려진 한계 / 후속 과제

  • Slack 양방향 동기화 — Webhook 한계로 thread 답글 동기화 불가. Bot Token 도입 시 Phase 6+에서 보강.
  • Blob staging cleanup — TTL 기반 cleanup cron 미구현. 운영 매뉴얼에 메모만. 실제 운영에서 1일 이상 매핑 안 된 staging 파일 삭제 필요.
  • 티켓 동시 접수 동시성 — 현재 retry 1회로 대응. 트래픽 높아지면 Postgres advisory lock 필요.
  • IC-09~IC-11 (AI 보조) — P2/P3. MVP 후 별도 진행.
  • IS-05 엑셀 다운로드 / IS-06 수동 발송 — Phase 6 보강.
  • 티켓 피드백 (⑦) — Phase 6에서 ticket_feedback 테이블 + 완료 상태 위젯.
  • ticket_form_fields 어드민 UI — 빈 테이블. Phase 9에서 본격 구현.
