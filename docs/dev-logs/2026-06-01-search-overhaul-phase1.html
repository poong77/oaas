1. 점검 결과 — 핵심 원인

"매칭은 동의어로 똑똑하게, 점수·정렬·하이라이트는 원본 글자로 멍청하게" 하고 있었음. 동의어로 찾아낸 결과("CI"→"체크인")가 제목에 원본 글자가 없다는 이유로 score 0점으로 가라앉아 관련도 정렬·뱃지·하이라이트에서 모두 누락되던 것이 "검색이 맘에 안 든다"의 주원인.
#결함상태
1동의어 매칭 결과가 원본 검색어 기준 점수라 0점 → 관련도 정렬 무력화수정
2점수가 구 필드 summary30s만 보고 신규 summary 미반영수정
3keywords 배열 매칭이 점수에 미반영수정
4하이라이트가 원본 검색어만 — 동의어 매칭 단어 표시 안 됨수정
5FAQ·공지 검색은 동의어 확장을 아예 안 함(도움말만 적용)수정
6다단어 입력 시 OR 매칭이라 정밀도 낮음Phase 2
7articles_search_tsv GIN 인덱스 생성만 하고 미사용(ILIKE 풀스캔)Phase 2

FAQ/공지 매칭원본 ILIKE만동의어 확장 후 term별 ILIKE OR (도움말과 동일)

3. 수정 파일

  • lib/text/search-match.ts — 신규. matchesAnyTerm, keywordsMatchAnyTerm, buildHighlightRegex 공용 헬퍼
  • lib/services/articles.ts — searchArticles 점수 로직을 확장 term 기준으로 교체, titleMatch 추가
  • lib/services/faqs.ts — searchFaqs에 동의어 확장 도입 + 점수/questionMatch
  • lib/services/notices.ts — searchNotices에 동의어 확장 도입 + 점수/titleMatch
  • app/search/page.tsx — 확장 term 하이라이트, 뱃지 플래그 연동, summary 우선 표시

4. 검증 / 주의사항

npm run typecheck 통과. 동작 검증은 실데이터 검색어("CI", "하우스키핑" 등)로 수동 확인 권장.
정렬은 여전히 DB에서 publishedAt DESC 100건을 가져온 뒤 앱에서 점수 재정렬. 데이터가 100건을 크게 넘기면 관련도 높은 옛 글이 누락될 수 있음 → Phase 2(tsvector/시맨틱)에서 DB 랭킹으로 이관 예정.

5. Phase 2 — 하이브리드 시맨틱 검색 (완료)

pgvector + OpenAI text-embedding-3-small(1536차원) 도입. searchArticles가 키워드 leg(ILIKE+동의어) + 벡터 leg(코사인 최근접)를 id로 병합하고 점수 = 키워드 점수 + cosine_sim × 4로 정렬. 키워드로 안 잡히는 의미 매칭("결제가 안돼요" → 결제 오류 문서)을 벡터 leg가 커버.
구성내용
스키마articles.embedding vector(1536) + HNSW 코사인 인덱스(articles_embedding_hnsw). 마이그레이션 0020 — CREATE EXTENSION vector 수동 보강(멱등)
임베딩 생성발행/수정 시 자동(generateArticleEmbedding · title+summary+keywords+body). 기존 데이터는 npm run db:backfill-embeddings
graceful degradeOPENAI_API_KEY 미설정/quota 오류 → 임베딩 null → 키워드 검색 폴백. 서비스 중단 없음, 키 활성 즉시 시맨틱 자동 ON
적용 절차db:migrate(완료) → 결제 정상화 → db:backfill-embeddings → 시맨틱 활성
