문제 1 — 비활성 분류를 가리키는 콘텐츠 (사용자에게 안 보임)

/help 랜딩·검색은 getProductCategories()가 활성 대분류만 노출 → 아래 코드의 콘텐츠는 어느 제품 카드에도 안 잡힘.

문제 2 — 신규 분류는 준비됐으나 콘텐츠 이관 누락

활성 계층에 들어갈 자리(pms_housekeeper·pms_webpos·etc_message·etc_hoteltv 등)가 이미 비어 있는 채로 존재 → 개편이 절반만 진행된 상태.

문제 3 — 설계상 정상 콘텐츠가 대분류 코드에만 매핑

아티클 product_code는 대분류만 사용하고 제품 내 내비게이션은 menu_taxonomies 라벨 트리로 구동. product_code를 하위 코드로 내리면 /help/[product] 라우팅이 깨지므로 의도된 설계로 판단 → 하위 코드 강제 이전은 제외.

⚠️ 남은 별개 항목 (미처리)

  • cms/homepage 메뉴트리 정합 — menu_taxonomies에 product 불일치 고아노드 7건(cms "홈페이지 …" 노드의 parent가 homepage 계열). 운영팀의 라이브 재분류와 연관된 사안으로, 편집 종료 후 별도 정리 권장.
  • 비활성 분류 보존 — web·config·oachat·lock·pms_pms_ver는 참조 0 확인 후에도 물리 삭제하지 않고 소프트 비활성 유지(이력 보존 원칙).
  • etc 대분류 카드 — 이제 콘텐츠 11+3건 보유로 정상 노출. 빈 카드 문제 해소.
  • 하위 33분류 콘텐츠 0건 — 접수폼 cascade 전용(의도된 설계). 핸드북 분류엔 미사용으로 정상.

🛠 작업 중 발견·교정한 리스크

운영 DB 직결 + sql 태그 즉시실행: connectPg().sql은 pool.query 기반으로 태그 호출 즉시 쿼리를 전송합니다. 최초 스크립트가 dry-run에서 UPDATE 문을 변수에 담기만 했는데도 menu_taxonomies UPDATE가 실제 전송되어 menu 이전이 dry-run 중 선반영됨. → 매핑이 의도한 결과와 동일하여 데이터 손상은 없었고, 스크립트를 즉시실행 안전 패턴으로 교정 후 articles/faqs 이전을 완료. BEGIN/COMMIT도 pool 특성상 트랜잭션이 성립하지 않아 제거(매핑 멱등으로 대체).
