요약

어드민 > 인사이트 > 검색로그에서 같은 검색어가 같은 초에 2~3건씩 찍히는 현상을 제보받아 원인을 추적했습니다. DB를 직접 점검한 결과, 검색 로깅(logSearch)이 /search 서버 컴포넌트의 렌더 부수효과로 실행되는데, Next 16의 <Link> 기본 prefetch가 호버/터치 시점에 페이지를 한 번 더 렌더하면서 검색 1회 → 로그 2건이 생기고 있었습니다. prefetch 스킵과 윈도우 dedup 2겹으로 막고, 부수적으로 측정 불가한 "세션 체류" 컬럼을 제거하고 유입일시에 초를 추가했습니다.

1. 원인 분석 — DB 실증

진단 스크립트(db/scripts/inspect-dup-search-logs.ts)로 중복 행을 ms 단위까지 확인.

검색어같은 초 내 간격세션키해석
키 발급 안됨58130ms → 58927ms (797ms)nullprefetch + 클릭
카드키 발급 실패49720 → 49840 (120ms)nullprefetch + 클릭
도어락24206 → 24905 (699ms)nullprefetch + 클릭

최근 3일 106건 중 10건이 중복. 모두 session_key=null + 같은 초 + 100~800ms 간격 → 사람의 재검색이 아니라 한 번의 검색에 서버 렌더가 2번 실행된 흔적.

근본 원인. app/search/page.tsx의 SearchResults(RSC, force-dynamic) 안에서 await logSearch(...)를 호출 — 즉 렌더할 때마다 INSERT. 인기 키워드 칩이 기본 prefetch <Link>(home-hero.tsx, search/page.tsx)라, 호버/터치 시 prefetch 렌더(로그 1) → 클릭 후 실제 렌더(로그 2). 검색 수 과대 → CTR 분모 부풀림 등 지표 왜곡.

2. 수정 — 2겹 방어

FIX ① prefetch 요청에서는 로깅 스킵 (근본 원인 제거)

prefetch 요청은 Next-Router-Prefetch: 1 / purpose: prefetch 헤더를 보냄. 서버 컴포넌트에서 감지해 스킵.

// app/search/page.tsx
let logId: string | null = null;
if (query && !(await isPrefetchRequest())) {
  const me = await getCurrentUser();
  logId = await logSearch({ query, counts, ... });
}

async function isPrefetchRequest(): Promise<boolean> {
  const h = await headers();
  return h.get('next-router-prefetch') === '1'
    || h.get('purpose') === 'prefetch'
    || h.get('x-purpose') === 'prefetch';
}

FIX ② 5초 윈도우 dedup (백스톱)

탭/필터 재렌더·근접 더블렌더 등 어떤 경로로도 중복이 안 생기도록. 기존 행 logId를 재사용해 클릭/전환 추적도 그대로 동작. SELECT → INSERT라 제약 충돌·예외 없음.

// lib/services/search-logs.ts — logSearch()
const since = new Date(Date.now() - 5_000);
const [recent] = await db.select({ id: searchLogs.id })
  .from(searchLogs)
  .where(and(
    gte(searchLogs.createdAt, since),
    eq(searchLogs.normalizedQuery, normalizedQuery),
    userId ? eq(...userId) : isNull(...),
    productCode ? eq(...) : isNull(...),
    sessionKey ? eq(...) : isNull(...),
  ))
  .orderBy(desc(searchLogs.createdAt)).limit(1);
if (recent?.id) return recent.id;  // 중복 INSERT 대신 재사용
배포 후 검증. 같은 진단 스크립트로 재확인 결과, 100~800ms 간격의 같은 초 쌍이 전부 사라짐(가장 가까운 게 6초). 특히 인기 칩 체크인 오류가 이전엔 같은 초 2건이었으나 이제 1건만 기록 → prefetch 스킵 정상 동작 확인. "도어락 오작동" 3건(04:06:28 / 04:07:00 / 04:07:06)은 6초·32초 간격의 실제 반복 검색으로, dedup 윈도우(5초)를 넘어 정상적으로 별개 기록된 것.

3. 부수 정리

REFACTOR 세션 체류 컬럼 제거

logSearch가 sessionKey 없이 기록 → dwellSeconds는 항상 null → 화면에 영원히 "—". 측정 불가 지표라 컬럼·관련 dead code를 정리.

  • list-client: 데스크탑 "세션 체류" 컬럼 + 모바일 "체류" 라인 + formatDwell 헬퍼 제거
  • search-logs.ts: SearchLogRow.dwellSeconds, LEAD 윈도우 dwellExpr, 매핑 제거

FEAT 유입일시 초(ss) 표시

같은 분 안 여러 검색의 순서·간격을 구분할 수 있도록 초까지 표시. 기존 formatDateTimeKst(티켓 등 공용)는 유지하고 신규 포맷터만 추가.

// lib/business-hours/format.ts
export function formatDateTimeSecKst(d: Date): string {
  return d.toLocaleString('ko-KR', {
    timeZone: 'Asia/Seoul', month:'long', day:'numeric', weekday:'short',
    hour:'2-digit', minute:'2-digit', second:'2-digit', hour12:false,
  });  // "6월 2일 (화) 13:07:06"
}
