## 1. 배경 (Why)

### 1.1 발견된 문제

- **외부 스펙 v1.1이 도착했으나 우리 프로젝트 8번 규칙(어드민 DB 편집 우선)과 정면 충돌** — 스펙은 `content/*.md` + `registries/*.yaml` 파일 전제. 매니저는 git을 쓰지 않음. Vercel은 런타임 파일 시스템 영속성 없음 → 콘텐츠 수정마다 빌드 트리거 불가.
- **기존 `articles` 스키마는 SS-02/03/06의 기본 골격만 — 검색·챗봇 친화적 메타가 부재**:
  - `content_type` (사용자 의도 분류) 없음 → RAG 답변 라우팅 불가
  - `keywords` (동의어·약어·UI 라벨) 없음 → `synonyms-master`로 확장하더라도 글마다 잡히지 않는 표현을 흡수할 장치 없음
  - `applies_to` (플랜/모델 적용 범위) 없음 → "이 기능 우리도 됨?" 질문에 답할 메타 없음
  - 명시적 `status` 컬럼 없음 (`published_at IS NULL`로 draft 표현) → draft 일괄 조회·필터·통계가 매번 NULL 비교 필요
- **매니저 에디터 미구현** — `articles` 스키마는 있으나 `/admin/articles` CRUD UI가 없음 (체크리스트 미완료 항목 `articles 스키마 + 어드민 편집 (SS-06, 마크다운 에디터)`).
- **URL 구조 미확정** — 현재 `/help/[product]/[slug]`. 스펙 §2.1의 3-값 `content_type` 도입 시 URL 패턴 결정 필요 (이미 Q-5에서 결정됨).
- **검색이 ILIKE만** — SS-01 통합 검색·매니저 티켓 검색 둘 다 `ILIKE '%keyword%'`. 동의어 사전(`synonyms-master`)이 도착 중이지만 **아티클의 `keywords` 컬럼**까지 인덱싱 대상에 포함해야 효과 최대화.
- **`related_article_ids`가 uuid[] — id 기반 안정 참조 원칙 위배** — 외부 스펙 §2의 핵심 가치(slug 기반 참조 = 표기 바뀌어도 안 깨짐)를 우리도 따를지 결정 필요.

## 5. DB 설계

### 5.1 `articles` 테이블 확장 (ALTER)

```ts
// db/schema/articles.ts (확장)
export const articleContentTypeEnum = pgEnum('article_content_type', [
  'howto',         // 사용방법 — 따라하기. title 동작형. H2: 목표/사전준비/단계/다음단계
  'feature',       // 기능설명 — 이해하기. title 명사구. H2: 개요/위치/항목설명/관련문서
  'troubleshoot',  // 문제해결 — 고치기. title 증상형. H2: 증상/원인/해결단계/그래도안되면
]);

export const articleStatusEnum = pgEnum('article_status', [
  'draft',
  'published',
]);

/** applies_to 형태 — JSONB. nullable. null = 전 고객/전 플랜 적용. */
export type ArticleAppliesTo = {
  feature?: string;        // feature-registry 키 (별도 마스터, P1 또는 후속)
  models?: string[];       // 디바이스 모델 id (devices 마스터, 후속)
};

export const articles = pgTable('articles', {
  ...commonColumns(),                                    // id (uuid), created_at, updated_at, is_active
  productCode: text('product_code').notNull(),            // 기존
  contentType: articleContentTypeEnum('content_type').notNull(),  // 🆕 P0-A
  status: articleStatusEnum('status').notNull().default('draft'), // 🆕 P0-A (published_at은 발행 시각 보관용으로 유지)
  categoryPath: text('category_path').array(),            // = menu_path. 기존 컬럼 그대로 활용
  slug: text('slug').notNull(),                           // 기존 — 글로벌 유일 유지
  title: text('title').notNull(),                         // 기존
  summary: text('summary'),                               // 🔁 summary30s 폐기 → summary (≤200자, Q-13)
  keywords: text('keywords').array().notNull().default(sql`'{}'`),  // 🆕 P0-A
  appliesTo: jsonb('applies_to').$type<ArticleAppliesTo>(),         // 🆕 P0-A, nullable = 전체 적용
  bodyMarkdown: text('body_markdown').notNull(),          // 기존 — Q-13에 따라 body로 리네임 검토
  toc: jsonb('toc').$type<TocEntry[]>(),                  // 기존 — 자동 추출 (P0-H)
  relatedSlugs: text('related_slugs').array().notNull().default(sql`'{}'`), // 🔁 relatedArticleIds(uuid[]) 폐기 → slug[] (Q-14)
  authorId: uuid('author_id').references(() => users.id, { onDelete: 'set null' }),
```

## 6. 검색 통합 (P0-U) — `synonyms-master` 결합

```ts
// app/search/page.tsx — 의사 코드
async function searchArticles(q: string, contentType?: ArticleContentType) {
  // 1. 입력 q 정규화 (lower, trim) + 토큰화
  const tokens = tokenize(q.toLowerCase().trim());

  // 2. synonyms-master로 확장
  const expanded = await Promise.all(tokens.map(expandKeywords));
  const allTerms = Array.from(new Set(expanded.flat()));

  // 3. articles 검색 — 3가지 매칭을 OR로 결합
  const results = await db.select().from(articles).where(and(
    eq(articles.status, 'published'),
    eq(articles.isActive, true),
    contentType ? eq(articles.contentType, contentType) : undefined,
    or(
      // (a) keywords 배열에 매칭 (GIN 인덱스)
      sql`${articles.keywords} && ${allTerms}`,
      // (b) tsvector 풀텍스트 매칭 (GIN 인덱스)
      sql`to_tsvector('simple', ${articles.title} || ' ' || coalesce(${articles.summary},'') || ' ' || ${articles.bodyMarkdown}) @@ to_tsquery('simple', ${allTerms.join(' | ')})`,
      // (c) ILIKE 폴백 (확장 안 됐을 때)
      sql`${articles.title} ILIKE ${'%' + q + '%'}`,
    ),
  )).limit(20);

  return results;
}
```

**핵심**: keywords 배열 매칭(GIN)과 tsvector 풀텍스트 매칭(GIN)을 OR로 결합. `synonyms-master`의 동의어 확장은 두 검색 모두에 적용. ILIKE는 안전망.

## 7. 본문 검증기 (P0-G)

```ts
// lib/articles/body-validator.ts
const REQUIRED_H2_BY_TYPE: Record<ArticleContentType, string[]> = {
  howto:        ['목표', '사전 준비', '단계', '다음 단계'],
  feature:      ['개요', '위치(메뉴 경로)', '항목 설명', '관련 문서'],
  troubleshoot: ['증상', '원인', '해결 단계', '그래도 안 되면'],
};

const NON_SELF_CONTAINED_PHRASES = ['위에서 말한', '앞서 설명한', '위에서 본', '앞에 나온'];

export function validateBody(markdown: string, contentType: ArticleContentType) {
  const errors: string[] = [];
  const warnings: string[] = [];

  const h2Headings = [...markdown.matchAll(/^##\s+(.+)$/gm)].map(m => m[1].trim());
  const requiredH2 = REQUIRED_H2_BY_TYPE[contentType];
  for (const req of requiredH2) {
    if (!h2Headings.some(h => h.includes(req.split('(')[0].trim()))) {
      errors.push(`필수 H2 누락: "${req}"`);
    }
  }

  for (const phrase of NON_SELF_CONTAINED_PHRASES) {
    if (markdown.includes(phrase)) {
      warnings.push(`자기완결 본문 위반: "${phrase}" — RAG 청크가 단독 노출되므로 본문에서 명시하세요`);
    }
  }

  return { errors, warnings };
}
```

저장 시 errors는 차단(draft 저장은 허용, published 전환은 차단). warnings는 표시만.

### Value Delivered (4-Perspective)

| Perspective | 내용 |
|:-|:-|
| **Problem** | 외부 스펙 v1.1은 RAG 친화적이나 우리 8번 규칙(어드민 DB 편집 우선)과 충돌. 기존 articles 스키마엔 content_type·keywords·applies_to·status 메타 부재. 매니저 에디터·검색 통합 모두 미구현. |
| **Solution** | 스펙의 스키마/검증/RAG 원칙은 차용하되 저장소를 Neon DB로 전환. articles에 4개 메타 컬럼 추가, menu_taxonomies/article_redirects 신설, 매니저 에디터 3페이지 + 메뉴 마스터 4페이지 신규, URL 패턴 재정렬 + 301 처리, synonyms-master 결합 검색 통합. |
| **Function UX Effect** | 매니저가 마크다운 메타 폼으로 아티클 작성·검증·발행. content_type 분류로 의도별 필터링. 호텔리어가 "CI" 검색해도 체크인 아티클 노출 (keywords + 동의어 + tsvector 3중 매칭). 메뉴 트리는 어드민이 DB 편집. |
| **Core Value** | RAG 친화 메타를 DB로 흡수해 검색·콘텐츠 운영 단일 진실 공급원 확보. 향후 챗봇/임베딩/채널.io 이관의 기반 인프라 완성. |
