## 3. DB 스키마 (확정안)

### 3.1 `articles` 테이블 확장

```ts
// db/schema/articles.ts — 확장 후
import {
  pgEnum, pgTable, text, jsonb, timestamp, uuid, integer, index, uniqueIndex,
} from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';
import { commonColumns } from './_shared';
import { users } from './users';

// 🆕 enum 2종
export const articleContentTypeEnum = pgEnum('article_content_type', [
  'howto',         // 사용방법 — 따라하기. title 동작형
  'feature',       // 기능설명 — 이해하기. title 명사구
  'troubleshoot',  // 문제해결 — 고치기. title 증상형
]);
export const articleStatusEnum = pgEnum('article_status', ['draft', 'published']);

// applies_to 형태 — null = 전 고객/전 플랜 적용 (Plan §2.4 권장 원칙)
export type ArticleAppliesTo = {
  feature?: string;        // feature-registry 키 (별도 마스터, 후속 Plan)
  models?: string[];       // 디바이스 모델 id (devices 마스터, 후속)
};

export type TocEntry = {
  level: 1 | 2 | 3;
  text: string;
  anchor: string;
};

export const articles = pgTable('articles', {
  ...commonColumns(),
  productCode: text('product_code').notNull(),                       // 기존
  contentType: articleContentTypeEnum('content_type').notNull(),     // 🆕
  status: articleStatusEnum('status').notNull().default('draft'),    // 🆕
  categoryPath: text('category_path').array(),                        // 기존 = menu_path
  slug: text('slug').notNull(),                                       // 기존, 전역 unique
  title: text('title').notNull(),                                     // 기존
  summary: text('summary'),                                           // 🔁 summary_30s → summary (Q-13)
  keywords: text('keywords').array().notNull().default(sql`'{}'`),    // 🆕
  appliesTo: jsonb('applies_to').$type<ArticleAppliesTo>(),           // 🆕 nullable
  bodyMarkdown: text('body_markdown').notNull(),                      // 기존 — markdown(+ inline HTML)
  toc: jsonb('toc').$type<TocEntry[]>(),                              // 기존, 발행 시 자동 추출
  relatedSlugs: text('related_slugs').array().notNull().default(sql`'{}'`), // 🔁 uuid → slug (Q-14)
  authorId: uuid('author_id').references(() => users.id, { onDelete: 'set null' }),
  lastEditorId: uuid('last_editor_id').references(() => users.id, { onDelete: 'set null' }), // 🆕
  publishedAt: timestamp('published_at', { withTimezone: true }),      // 기존 — published 전환 시만 set
  viewCount: integer('view_count').notNull().default(0),               // 기존
  helpfulYes: integer('helpful_yes').notNull().default(0),             // 기존
  helpfulNo: integer('helpful_no').notNull().default(0),               // 기존
}, (table) => [
  uniqueIndex('articles_slug_uq').on(table.slug),
  // 🆕 product + content_type + status 조합 빈번 조회 (목록·필터)
  index('articles_product_ct_status_idx').on(
    table.productCode, table.contentType, table.status, table.publishedAt,
  ),
  index('articles_status_published_idx').on(table.status, table.publishedAt),
  // 🆕 keywords 배열 OR 매칭 — synonyms-master expandKeywords와 결합
  index('articles_keywords_gin').using('gin', table.keywords),
  // 🆕 풀텍스트 — title + summary + body_markdown 통합
  // 표현식 인덱스는 drizzle 0.36에서 raw SQL로 처리 (마이그레이션에서 직접 CREATE)
]);

export type Article = typeof articles.$inferSelect;
export type NewArticle = typeof articles.$inferInsert;
```

**비고**:
- `bodyMarkdown` 컬럼명 유지 (Q-15). 내용은 "markdown + 인라인 HTML" hybrid (§7 참고)
- `summary` 길이 제한은 DB가 아닌 Zod에서 강제 (≤2000자 권장, 워닝)

### 3.2 `article_redirects` 신규 테이블 (P0-E)

```ts
// db/schema/article-redirects.ts
import { pgTable, text, uniqueIndex } from 'drizzle-orm/pg-core';
import { commonColumns } from './_shared';

export const articleRedirects = pgTable('article_redirects', {
  ...commonColumns(),
  fromPath: text('from_path').notNull(),
  toSlug: text('to_slug').notNull(),
  reason: text('reason'),  // 'content_type_change' | 'slug_rename' | 'manual'
}, (table) => [
  uniqueIndex('article_redirects_from_uq').on(table.fromPath),
]);
```

- `fromPath`는 path-only (host 미포함). 예: `/help/oa-pms/reservation-create`
- `toSlug`만 저장 (content_type/product는 articles에서 조회). slug가 unique이므로 충분
- 발행 후 content_type 변경 시 Server Action에서 자동 INSERT

### 3.3 인덱스 트랜스액션 안전성

- `articles_keywords_gin` 와 풀텍스트 GIN은 거대해질 수 있으므로 **CREATE INDEX CONCURRENTLY**로 분리 마이그레이션 권장 (운영 데이터 늘었을 때 잠금 회피). 본 Phase는 데이터 거의 없음 → 일반 CREATE INDEX로 충분.

### 4.2 `articles` ALTER — drizzle 산출 + 수동 보강

drizzle-kit이 자동 생성하는 부분은 컬럼 추가/타입 변환까지. 다음은 **마이그레이션 SQL 파일에 수동으로 추가**해야 함:

```sql
-- (drizzle 자동)
ALTER TABLE "articles" ADD COLUMN "content_type" "article_content_type" NOT NULL DEFAULT 'howto';
ALTER TABLE "articles" ADD COLUMN "status" "article_status" NOT NULL DEFAULT 'draft';
ALTER TABLE "articles" ADD COLUMN "keywords" text[] NOT NULL DEFAULT '{}';
ALTER TABLE "articles" ADD COLUMN "applies_to" jsonb;
ALTER TABLE "articles" ADD COLUMN "last_editor_id" uuid;
ALTER TABLE "articles" ADD COLUMN "related_slugs" text[] NOT NULL DEFAULT '{}';
ALTER TABLE "articles" ADD CONSTRAINT "articles_last_editor_id_users_id_fk"
  FOREIGN KEY ("last_editor_id") REFERENCES "users"("id") ON DELETE SET NULL ON UPDATE NO ACTION;
ALTER TABLE "articles" ADD COLUMN "summary" text;

-- (수동) 1) summary_30s → summary 데이터 이관 (혹시 운영 데이터 있을 경우)
UPDATE "articles" SET "summary" = "summary_30s" WHERE "summary_30s" IS NOT NULL;

-- (수동) 2) published_at IS NOT NULL → status='published' 마이그레이션
UPDATE "articles" SET "status" = 'published' WHERE "published_at" IS NOT NULL;

-- (수동) 3) related_article_ids(uuid[]) → related_slugs(text[]) 매핑
-- 운영 데이터 0건이라면 생략. 있다면:
UPDATE "articles" SET "related_slugs" = (
  SELECT COALESCE(array_agg(a2.slug), '{}'::text[])
  FROM unnest("articles"."related_article_ids") AS rel_id
  LEFT JOIN "articles" a2 ON a2."id" = rel_id
);

-- (수동) 4) deprecated 컬럼 DROP — Do 단계 직후 별도 안전 마이그레이션
-- (본 Phase 검증 후 다음 PR에서 분리 실행)
-- ALTER TABLE "articles" DROP COLUMN "summary_30s";
-- ALTER TABLE "articles" DROP COLUMN "related_article_ids";

-- (수동) 5) GIN 인덱스 추가
CREATE INDEX "articles_keywords_gin" ON "articles" USING gin ("keywords");
CREATE INDEX "articles_product_ct_status_idx" ON "articles"
  ("product_code", "content_type", "status", "published_at");
CREATE INDEX "articles_status_published_idx" ON "articles" ("status", "published_at");

-- (수동) 6) 풀텍스트 — 표현식 인덱스. simple config (한국어 형태소 미사용)
CREATE INDEX "articles_search_tsv" ON "articles"
  USING gin (to_tsvector('simple',
    coalesce("title", '') || ' ' ||
    coalesce("summary", '') || ' ' ||
    coalesce("body_markdown", '')
  ));

-- (수동) 7) default 제거 (앱이 명시적으로 채우도록)
ALTER TABLE "articles" ALTER COLUMN "content_type" DROP DEFAULT;
```

### 4.3 마이그레이션 안전 절차 (D-1)

1. `npm run db:generate` — drizzle 자동 산출 (컬럼 추가만)
2. 생성된 SQL 파일에 위 §4.2 수동 보강 SQL 추가
3. dev DB에 `npm run db:migrate` 실행 → 검증
4. preview 배포 → 발견된 회귀 수정
5. main → prod migrate

**롤백 전략**: deprecated 컬럼(`summary_30s`, `related_article_ids`)을 즉시 DROP하지 않고 한 사이클 후에 별도 마이그레이션으로 제거. 그 사이 회귀 발견 시 컬럼 그대로 살아 있음.

## 5. URL 라우팅 + 301 리다이렉트

### 5.1 신규 라우트

| URL | 페이지 | 데이터 |
|:-|:-|:-|
| `/help` | 통합 도움말 인덱스 | 6개 제품 카드 + content_type 필터 칩 (확장) |
| `/help/[product]` | 제품별 인덱스 | menu_taxonomies 트리 + 인기 아티클 |
| `/help/[product]/[content_type]/[slug]` | **신규 상세** | SS-03 6개 영역 |
| `/help/[product]/[slug]` | **레거시 → 301** | articles 조회 → content_type 발견 → 새 URL로 308 영구 리다이렉트 |

### 5.2 리다이렉트 로직 (`/help/[product]/[slug]/page.tsx`)

```ts
export default async function LegacyArticleRedirect({ params }: { params: Promise<{ product: string; slug: string }> }) {
  const { product, slug } = await params;
  const article = await getArticleBySlug(slug);
  if (!article) {
    // article_redirects 폴백 조회
    const redirect = await getArticleRedirectByFromPath(`/help/${product}/${slug}`);
    if (redirect) {
      const target = await getArticleBySlug(redirect.toSlug);
      if (target) {
        permanentRedirect(`/help/${target.productCode}/${target.contentType}/${target.slug}`);
      }
    }
    notFound();
  }
  permanentRedirect(`/help/${article.productCode}/${article.contentType}/${article.slug}`);
}
```

- `permanentRedirect` (Next.js 15 App Router) = HTTP 308 (영구). SEO/북마크에 적절
- article_redirects는 slug 자체가 바뀌었을 때 (rename) 백업 폴백
- 발행 후 content_type/slug 변경 시 Server Action이 article_redirects에 자동 INSERT

### 5.3 슬러그 변경 vs content_type 변경

- **content_type 변경**: slug는 유지 → 새 content_type 경로로 자연 리다이렉트 (articles 한 row 조회로 해결, redirects 테이블 불필요)
- **slug 변경**: 옛 slug로는 articles 못 찾음 → article_redirects 테이블 필요 (수동 INSERT 또는 자동 트래킹)

### 6.2 본문 검증기 (`lib/articles/body-validator.ts`)

```ts
const REQUIRED_H2_BY_TYPE = {
  howto:        ['목표', '사전 준비', '단계', '다음 단계'],
  feature:      ['개요', '위치(메뉴 경로)', '항목 설명', '관련 문서'],
  troubleshoot: ['증상', '원인', '해결 단계', '그래도 안 되면'],
} as const;

const NON_SELF_CONTAINED = ['위에서 말한', '앞서 설명한', '위에서 본', '앞에 나온'];

export type ValidationResult = {
  errors: string[];   // published 차단
  warnings: string[]; // 저장 허용, UI 표시만
};

export function validateBody(content: string, contentType: ArticleContentType): ValidationResult {
  const errors: string[] = [];
  const warnings: string[] = [];

  // §7에서 결정: 본문은 markdown + 인라인 HTML. ## H2는 markdown 그대로
  const h2Headings = [...content.matchAll(/^##\s+(.+)$/gm)].map(m => m[1].trim());

  for (const req of REQUIRED_H2_BY_TYPE[contentType]) {
    // 괄호 부분(예: "위치(메뉴 경로)")은 핵심 키워드만 비교
    const core = req.split('(')[0].trim();
    if (!h2Headings.some(h => h.includes(core))) {
      errors.push(`필수 H2 누락: "${req}"`);
    }
  }

  for (const phrase of NON_SELF_CONTAINED) {
    if (content.includes(phrase)) {
      warnings.push(`자기완결 본문 위반: "${phrase}" — RAG 청크 독립 노출 원칙 위반`);
    }
  }

  // 추가 워닝: title이 "및", "+", ","로 여러 작업 묶음 의심 (저장 시 title도 함께 전달)
  // → 별도 함수 validateTitle()로 분리

  return { errors, warnings };
}
```

```ts
},
```

**문제 진단**: 주석에 "tiptap-markdown serializer는 표준 마크다운만 지원하여 스타일 손실됨"이라 되어 있으나, 이는 **`html: false`일 때만 사실**. `html: true`로 설정되어 있으면 markdown serializer가 폰트사이즈/색상/정렬 등을 인라인 HTML 태그로 보존한 채 직렬화한다.

**증거**: `lib/editor/tiptap-types.d.ts:11`에 `editor.storage.markdown.getMarkdown()` 타입 선언 존재. `rich-editor.tsx:188`에서 이미 사용 중 (draft 복원 비교용).

### 7.2 적용안 (3가지 옵션 — D-2 결정 필요)

#### Option A — RichEditor 전체를 markdown 저장으로 회귀 (권장)

```ts
// rich-editor.tsx
onUpdate: ({ editor }) => {
  // ✅ markdown + 인라인 HTML 출력 — 폰트/색/정렬 등은 HTML 태그로 보존됨
  const md = editor.storage.markdown.getMarkdown();
  valueRef.current = md;
  onChange(md);
},
```

**장점**:
- 단일 출력 경로 — articles/notices/faqs/tickets 등 RichEditor 사용처 10곳 전부 markdown 통일
- `markdown-to-html.ts`의 HTML 감지 휴리스틱(`/^\s*<[a-zA-Z]/`) 불필요해짐
- SES 이메일·SMS 변환이 마크다운 기준으로 일관

**단점**:
- 회귀 위험 — commit 48ded32에서 HTML로 전환한 이유가 "풀스택 톨바 스타일 보존"이었음. `html: true`가 실제로 모든 케이스를 보존하는지 회귀 테스트 필요
- 기존 HTML 저장된 row가 있다면 한 번 마이그레이션 필요 (없으면 PR 단일 변경)

**회귀 검증 매트릭스** (D-2-A 진입 시 필수):

| 스타일 | markdown 표현 가능? | html:true로 보존되는가? |
|:-|:-:|:-:|
| H1/H2/H3 | ✅ | ✅ |
| 볼드/이탤릭/취소선 | ✅ | ✅ |
| **밑줄 (underline)** | ❌ | ✅ (`<u>`) |
| **형광펜 (highlight)** | ❌ | ✅ (`<mark>`) |
| 표 (table) | ✅ GFM | ✅ |
| 체크리스트 | ✅ GFM | ✅ |
| 이미지 | ✅ | ✅ |
| 링크 (target/rel) | ✅ + HTML | ✅ |
| **폰트 사이즈** | ❌ | ✅ (`<span style="font-size:...">`) |
| **폰트 컬러** | ❌ | ✅ (`<span style="color:...">`) |
| **폰트 패밀리** | ❌ | ✅ |
| **정렬 (좌/중/우/양쪽)** | ❌ | ✅ (`<p style="text-align:...">`) |
| YouTube 임베드 | ❌ | ✅ (커스텀 HTML) |
| 코드블록 | ✅ | ✅ |

→ markdown으로 표현 불가능한 7종이 `html: true` 옵션으로 인라인 HTML로 보존됨. tiptap-markdown 0.9.0 검증 필요.

#### Option B — articles만 markdown 저장, 다른 사용처(notices/tickets 등)는 HTML 유지

- RichEditor에 `outputFormat: 'markdown' | 'html'` prop 추가
- articles 에디터만 'markdown' 지정
- 다른 사용처는 'html' default 유지 (회귀 위험 0)

**장점**: 회귀 격리. articles만 검증하면 됨.
**단점**: 코드 분기. 유지보수 복잡. body-validator도 분기 필요.

#### Option C — articles도 HTML 저장 + 검증기를 HTML→markdown 변환 후 검증

- body 컬럼은 HTML 그대로
- body-validator 진입 시 `html-to-markdown` 변환 (turndown 등) 후 검증
- 저장은 HTML, 검증/RAG 색인은 markdown

**장점**: 에디터 회귀 0
**단점**: 변환 비용 매 검증마다 발생. Q-15 결정(body_markdown 유지)과 어긋남

### 7.3 권장: Option A + 회귀 가드

본 Design 권장은 **Option A**. 단, 다음 가드:

1. **회귀 검증 페이지** — `/admin/sandbox/editor-check`에 위 §7.2 매트릭스 14종 테스트 케이스 모두 입력 → markdown 출력 → 다시 입력 → 결과 시각 비교
2. **마이그레이션 무결성** — 기존 HTML 저장된 row가 있다면 markdown 컬럼명 그대로 둔 채로 HTML 저장 허용 (markdown-to-html.ts의 휴리스틱 그대로 유지). 즉, **읽기는 hybrid 허용, 쓰기만 markdown로 통일**
3. **격리 PR** — RichEditor 변경은 articles 에디터 PR과 분리해서 별도 PR로 진행. 회귀 발견 시 RichEditor만 revert 가능

### 7.4 body-validator 입력 형태

§6.2 validateBody는 markdown 입력 가정. `## H2`는 markdown 표기. Option A로 가면 검증기 그대로 작동. Option B/C는 검증기 분기 필요.

→ **본 Design 본문은 Option A 채택 가정.** D-2에서 결정 후 §6.2 수정 가능.

## 11. 검색 통합 (synonyms-master + tsvector + keywords)

### 11.1 통합 검색 의사 코드 (P0-U)

```ts
// app/search/page.tsx (Server Component)
async function searchArticles(q: string, contentType?: ArticleContentType) {
  const tokens = tokenize(q.toLowerCase().trim()); // 공백/구두점 분리
  const expanded = await Promise.all(tokens.map(expandKeywords)); // synonyms-master
  const allTerms = Array.from(new Set(expanded.flat()));

  if (allTerms.length === 0) return [];

  const tsQuery = allTerms.map(t => t.replace(/['"\\]/g, '')).join(' | ');

  const rows = await db.select().from(articles).where(and(
    eq(articles.status, 'published'),
    eq(articles.isActive, true),
    contentType ? eq(articles.contentType, contentType) : undefined,
    or(
      // 1. keywords 배열 매칭 (GIN: articles_keywords_gin)
      sql`${articles.keywords} && ${allTerms}::text[]`,
      // 2. 풀텍스트 (GIN: articles_search_tsv)
      sql`to_tsvector('simple',
        coalesce(${articles.title},'') || ' ' ||
        coalesce(${articles.summary},'') || ' ' ||
        coalesce(${articles.bodyMarkdown},'')
      ) @@ to_tsquery('simple', ${tsQuery})`,
      // 3. ILIKE 폴백 (확장 실패 시)
      sql`${articles.title} ILIKE ${'%' + q + '%'}`,
    ),
  )).orderBy(desc(articles.publishedAt)).limit(20);

  return rows;
}
```

### 11.2 매니저 티켓 검색 → 관련 아티클 추천 (P0-V)

매니저가 티켓 상세에서 답변 작성 시, 본문/title의 token을 synonyms-master로 확장 → articles.keywords 매칭 → 사이드 패널에 후보 아티클 목록 노출 (클릭 시 마크다운 인용 삽입).
